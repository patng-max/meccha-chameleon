## Problem Statement

Meccha Chameleon has a solid M1 foundation: factions, privacy architecture, Supabase Auth, RLS, H3 utilities, and a static landing page. The next product slice needs to turn that foundation into the first real player loop:

- A visitor signs in with GitHub, becomes a player, chooses a faction, sets a display name, and reaches a dashboard.
- The dashboard shows the Reading pilot as an actual faction-control surface, not a static preview.
- The browser gets only public, game-safe state: faction identity, display names, aggregate stats, H3 cell polygons, labels, and territory state.
- The API layer becomes the only application boundary for player creation, dashboard data, and territory map payloads.
- The RLS model is tightened before more browser reads make privacy bugs harder to unwind.
- Tests start now, because later milestones add uploads, exact GPS, moderation, and proof media.

## What Makes This Cool

The cool version is not "log in and see a map." The cool version is that joining a faction immediately changes how the city looks. A new player should land on a dashboard where Reading feels alive: cells have owners, weak points, pending activity, and faction momentum. The map is the game board, but it never becomes a treasure map with pins.

The product tension is useful: the player wants enough signal to hunt, compete, and feel faction ownership; the system must never reveal exact coordinates. M2/3 should make that tension visible as a feature, not a disclaimer.

## Constraints

- Exact coordinates remain server-only. `private_hide_locations.exact_location` must not appear in browser JavaScript, API responses, logs, map markers, pins, GeoJSON points, or browser Supabase queries.
- Public map payloads can contain H3 cell IDs, H3-derived polygon boundaries, approximate labels, controller faction, counts, and state.
- Browser Supabase client must use only the anon key and should not query private tables.
- `SUPABASE_SERVICE_ROLE_KEY` and `TURNSTILE_SECRET_KEY` remain server-only and unprefixed.
- Existing M1 documents in `docs/` are canonical and must not be contradicted.
- M2/3 should not implement hide placement, proof upload, moderation workflow, or territory recalculation jobs yet. It can expose state shaped for those later flows.

## Premises

1. Player identity is the correct first unlock. A faction game without committed player identity cannot produce meaningful territory, stats, or social motivation.
2. The dashboard is the right first authenticated surface. It gives players a reason to complete onboarding before hide placement exists.
3. The Reading map should ship with real API-shaped data even if the underlying territory data is seeded or simple at first.
4. The map should show H3 cells only. No marker-based fallback is acceptable because it teaches the wrong product pattern.
5. The player RLS privacy fix belongs in this slice. Once authenticated browser views expand, broad public `players` reads become harder to reason about.
6. The test foundation belongs in this slice. Auth, RLS, and map privacy are now product behavior, not just implementation detail.

## Approaches Considered

### Approach A: Minimal Auth-to-Dashboard Slice

Summary: Implement GitHub OAuth completion, player creation/update, faction selection, display name, and a basic dashboard that renders API-provided faction stats plus a static H3 cell dataset.

Effort: M
Risk: Low

Pros:
- Ships the fastest usable authenticated product loop.
- Keeps MapLibre complexity lower by using a small Reading pilot dataset.
- Establishes route protection, onboarding states, and API conventions early.

Cons:
- Territory may still feel partially staged if map data is too static.
- Does not fully exercise database-derived territory queries.
- Risks deferring too much of the map data model until later.

Reuses:
- Existing Supabase SSR auth callback.
- Existing faction data and H3 utilities.
- Existing M1 RLS helpers and docs.

### Approach B: Product-Complete M2/3 Foundation

Summary: Build onboarding, dashboard, real API route boundaries, MapLibre Reading H3 cells, RLS adjustments for player privacy, and the first Vitest/Playwright skeleton focused on auth, RLS, and map payload safety.

Effort: L
Risk: Medium

Pros:
- Creates the right architecture before hide submission and proof flows add sensitive data.
- Makes the dashboard feel like the actual game board, not a placeholder.
- Gives future milestones stable contracts: `/api/player`, `/api/dashboard`, `/api/territory`.

Cons:
- More files change at once.
- Requires careful distinction between public player-facing data and private auth-owned data.
- MapLibre SSR/client boundaries need attention in Next.js.

Reuses:
- `src/lib/supabase/server.ts` for authenticated server reads/writes.
- `src/lib/h3-utils.ts` for cell boundary conversion.
- `territory_events`, `public_hides`, and `players` schema.
- Existing design language from the landing page.

### Approach C: Map-First Public Spectator Mode

Summary: Build the Reading MapLibre map first as a public route, then add sign-in and onboarding after the public game board is compelling.

Effort: M
Risk: Medium

Pros:
- Produces the most immediately visible demo.
- Lets non-authenticated visitors understand the faction war before joining.
- De-risks MapLibre rendering in isolation.

Cons:
- Delays the M2 identity milestone.
- Can accidentally optimize for spectators over players.
- Public-first map work increases privacy review burden before the API layer is settled.

Reuses:
- Existing static Reading cells and landing page preview.
- H3 utilities and faction styling.

## Recommended Approach

Choose Approach B: Product-Complete M2/3 Foundation.

This is the right lake to boil. The extra work over Approach A is not polish; it is the structural product surface that prevents the next milestones from stacking sensitive features on a weak boundary. The complete slice should include:

1. Authenticated player lifecycle:
   - GitHub OAuth callback lands the user in an onboarding state if no `players` row exists.
   - Faction select uses the three existing factions.
   - Display name validation is explicit: short, non-empty, visible to other players, and not treated as a secret.
   - Completed players reach `/dashboard`.

2. Dashboard:
   - Player summary: display name, faction badge, joined date or activity.
   - Faction standings: cells controlled, active hides, contested cells.
   - Reading territory map as the main dashboard surface.
   - Empty and seed states that still feel intentional: "No live hides yet" should become a founding opportunity, not a blank table.

3. API layer:
   - `GET /api/player/me`: authenticated player state and onboarding status.
   - `POST /api/player/onboard`: faction and display name creation/update, auth required, Turnstile protected.
   - `GET /api/dashboard`: authenticated dashboard summary, no exact coordinates.
   - `GET /api/territory`: public/auth-safe Reading cell state, no exact coordinates, no markers, no point GeoJSON.

4. MapLibre territory:
   - Client-only MapLibre component.
   - GeoJSON polygons generated from H3 cell boundaries, with `[lat, lng]` reversed to `[lng, lat]` and polygons explicitly closed.
   - Fill and border styles by controller state: Verdant, Ember, Tide, Unclaimed, Contested.
   - Popup content limited to approximate label, controller, public state, and aggregate counts.

5. Player RLS privacy fix:
   - Treat `players` as partly public identity, not a dumping ground.
   - Public reads should expose display identity needed for gameplay, but not auth-owned or future sensitive fields.
   - Prefer a public view such as `public_player_profiles` for display name/faction and restrict direct `players` reads to owner/service role, or document why the existing table policy is acceptable before expanding use.

6. Test foundation:
   - Vitest setup for H3, game data, and API contract helpers.
   - RLS-oriented integration tests or SQL policy checks for `players`, `private_hide_locations`, `public_hides`, and `territory_events`.
   - Playwright skeleton for auth redirect, onboarding page rendering, dashboard protection, and map payload privacy.
   - CI commands for lint, type-check, and tests.

## Open Questions

- Should display names be globally unique, faction-scoped unique, or non-unique? Recommendation: globally unique enough for gameplay clarity, enforced case-insensitively.
- Should faction choice be permanent for the pilot? Recommendation: lock it for M2/3; add moderator/admin reset later if needed.
- Should `/api/territory` be public or authenticated-only at first? Recommendation: public-safe by design, but dashboard can be authenticated-only. The payload must be safe either way.
- Is the Reading pilot seeded with real `territory_events`, derived from `public_hides`, or backed by static seed data until M4? Recommendation: use DB-shaped seed data through the API so the frontend contract does not change.
- Should player profile public data use a SQL view or API-only projection? Recommendation: SQL view plus API projection gives the strongest defense-in-depth.

## Success Criteria

- A signed-out visitor who tries `/dashboard` is redirected to `/`.
- A signed-in GitHub user with no player row is sent to onboarding.
- A signed-in user can choose Verdant, Ember, or Tide, set a display name, and reach the dashboard.
- A completed player sees faction identity, dashboard stats, and the Reading territory map.
- The territory map renders H3 polygons and no markers or point GeoJSON.
- Browser network responses do not include `exact_location`, `latitude`, `longitude`, `ST_X`, `ST_Y`, or private table data.
- Browser code does not query `private_hide_locations`.
- Player RLS behavior is explicitly tested or documented with a migration-level fix.
- The first test commands exist and run locally.
