## Architecture Decision

Use API-first application boundaries for M2/3.

Browser components should not query gameplay tables directly except through narrow server APIs. The browser Supabase client can keep handling auth session state, but gameplay reads and writes should go through route handlers that shape responses explicitly.

Recommended routes:

- `GET /api/player/me`
- `POST /api/player/onboard`
- `GET /api/dashboard`
- `GET /api/territory`

Keep `/api/territory` public-safe by design even if the first consumer is authenticated dashboard. A public-safe contract prevents a later route protection mistake from becoming a privacy bug.

### Required Changes

1. Add uniqueness and normalization support for player identity:

```sql
alter table public.players
  alter column user_id set not null;

create unique index players_user_id_key
  on public.players (user_id);

create unique index players_display_name_lower_key
  on public.players (lower(display_name));
```

Recommendation: display names are globally unique and case-insensitive for the pilot. This prevents confusing faction identity in chat, leaderboards, and moderation later.

2. Replace broad `players` public read with a public profile view:

```sql
create view public.public_player_profiles as
select
  id,
  faction,
  display_name,
  created_at
from public.players;
```

Then adjust RLS:

- Direct `players` SELECT: owner or service role only.
- `public_player_profiles`: grant select to anon/authenticated if Supabase view permissions require it.

Rationale: today `players` contains `user_id` and `last_active_at`. That is not an exact-coordinate leak, but it is unnecessary public identity metadata. Future columns could become sensitive if direct public reads remain the default.

3. Add territory cell materialization for map payloads.

Option A: derive from latest `territory_events` at request time.  
Option B: add a `territory_cells` table maintained by future jobs.

Recommendation: add `territory_cells` now and seed it for Reading.

```sql
create table public.territory_cells (
  h3_cell text primary key,
  area_label text not null,
  controller_faction public.faction_id,
  state text not null default 'unclaimed'
    check (state in ('unclaimed', 'controlled', 'contested')),
  active_hide_count integer not null default 0 check (active_hide_count >= 0),
  contested_hide_count integer not null default 0 check (contested_hide_count >= 0),
  updated_at timestamptz not null default now()
);

alter table public.territory_cells enable row level security;

create policy "territory cells are readable by everyone"
  on public.territory_cells for select
  using (true);

create policy "service role manages territory cells"
  on public.territory_cells for all
  using (auth.role() = 'service_role')
  with check (auth.role() = 'service_role');
```

Why not derive everything from `territory_events` now:

- M2/3 needs a stable map contract before real recalculation jobs exist.
- A materialized table gives `/api/territory` simple bounded reads.
- M6 can update this table from the official recalculation job without changing frontend contracts.

4. Keep `territory_events` append-only.

Current policy allows service role insert and public select. Do not add update/delete policies.

5. Consider hardening `moderator_actions` append-only in this migration or explicitly defer it.

The current security model already notes that moderator actions allow update/delete through a broad `for all` policy. M2/3 does not use moderation, so this can be deferred, but it should be tracked before M4.

## API Contracts

All responses must be explicit JSON objects with no table passthrough.

### `GET /api/player/me`

Auth: required.  
Turnstile: not required because GET.  
Server client: authenticated SSR anon client is sufficient.

Response when no player row exists:

```json
{
  "status": "needs_onboarding",
  "player": null
}
```

Response when onboarded:

```json
{
  "status": "ready",
  "player": {
    "id": "uuid",
    "displayName": "MossRunner",
    "faction": "verdant",
    "createdAt": "2026-06-29T00:00:00.000Z"
  }
}
```

Never return `user_id` or `last_active_at`.

### `POST /api/player/onboard`

Auth: required.  
Turnstile: required by middleware.  
Server client: authenticated SSR anon client should work with owner insert/update policies. Service role is not necessary.

Input:

```json
{
  "displayName": "MossRunner",
  "faction": "verdant",
  "turnstileToken": "token-from-client"
}
```

Validation:

- `displayName`: trim, 3 to 24 characters.
- Allowed characters: letters, numbers, space, underscore, hyphen.
- Reject reserved or misleading names if a simple list exists; otherwise defer.
- `faction`: one of `verdant`, `ember`, `tide`.
- Faction is locked after first creation for M2/3.

Behavior:

- If no row exists, insert.
- If row exists but display name changes, allow update.
- If row exists and faction differs, reject with 409.
- Case-insensitive duplicate display name returns 409.

Response:

```json
{
  "status": "ready",
  "player": {
    "id": "uuid",
    "displayName": "MossRunner",
    "faction": "verdant"
  }
}
```

### `GET /api/dashboard`

Auth: required.  
Turnstile: not required because GET.  
Server client: authenticated SSR anon client for player row; service role is not needed if public aggregates are readable.

Response:

```json
{
  "player": {
    "id": "uuid",
    "displayName": "MossRunner",
    "faction": "verdant"
  },
  "factions": [
    {
      "id": "verdant",
      "name": "Verdant",
      "controlledCells": 12,
      "activeHides": 4,
      "contestedCells": 2
    }
  ],
  "map": {
    "center": [-1.0, 51.0],
    "zoom": 12
  }
}
```

The sample values above are shape examples only. The implementation must use the project constants for Reading center/zoom if they exist, or define them in a server-safe game config module.

### `GET /api/territory`

Auth: optional.  
Turnstile: not required because GET.  
Server client: anon server client or service role only if needed for seed access. Prefer public RLS-readable `territory_cells`.

Response:

```json
{
  "type": "FeatureCollection",
  "features": [
    {
      "type": "Feature",
      "id": "h3-cell-id",
      "properties": {
        "h3Cell": "h3-cell-id",
        "areaLabel": "Reading centre",
        "controllerFaction": "verdant",
        "state": "controlled",
        "activeHideCount": 2,
        "contestedHideCount": 0
      },
      "geometry": {
        "type": "Polygon",
        "coordinates": [[[0, 0], [0, 0], [0, 0], [0, 0]]]
      }
    }
  ]
}
```

Privacy rule: no point geometries, markers, exact coordinates, private table ids, `private_location_id`, `player_id`, clue text, clue photo URLs, or capture claim data.

Use H3 boundaries only. The GeoJSON coordinate pairs are polygon vertices, not hide locations.

## Map Implementation

Install dependencies:

- `maplibre-gl`
- optionally `@types/maplibre-gl` only if required by the installed package version

Recommended component structure:

- `src/app/dashboard/page.tsx`: server component, redirects if signed out or not onboarded.
- `src/app/onboarding/page.tsx`: client form or server shell plus client form.
- `src/components/map/territory-map.tsx`: client-only MapLibre component.
- `src/components/map/territory-map.module.css`: stable sizing and responsive layout.
- `src/lib/territory/geojson.ts`: H3 cell to GeoJSON conversion.
- `src/lib/territory/contracts.ts`: zod schemas and TypeScript types for API payloads.

MapLibre rules:

- The component must be marked `"use client"`.
- Dynamically import MapLibre or isolate it in the client component so SSR never touches `window`.
- Import `maplibre-gl/dist/maplibre-gl.css` from a client-safe location or global CSS if Next allows it.
- Render one GeoJSON source with fill and line layers.
- Use fill color expressions based on `controllerFaction` and `state`.
- Popups show only `areaLabel`, controller, state, active hide count, and contested count.
- Do not add marker layers.
- Do not add point GeoJSON.
- Close H3 polygons by repeating the first coordinate at the end.
- Convert `h3-js` boundary output from `[lat, lng]` to GeoJSON `[lng, lat]`.

Performance guardrails:

- Keep the Reading pilot seed modest for M3.
- Use one source/layer pair instead of one React component per cell.
- Avoid storing map object in React state; use refs.
- Clean up the MapLibre instance on unmount.

## Auth and Routing

Recommended flow:

1. Signed-out visitor clicks GitHub sign-in.
2. Supabase callback establishes session.
3. Callback redirects to `/dashboard`.
4. `/dashboard` server page checks session.
5. If no session, redirect to `/`.
6. If session exists but no player row, redirect to `/onboarding`.
7. Onboarding POST creates player and redirects to `/dashboard`.

Middleware:

- Preserve Turnstile on POST `/api/*`.
- Do not require Turnstile for OAuth callback GET routes.
- Do not gate every dashboard GET with Turnstile; session auth is enough for page loads.

Server clients:

- Use `createServerClient()` for session-aware reads/writes owned by the current user.
- Use `createServiceRoleClient()` only for administrative operations or seed/maintenance tasks.
- Do not use the service role in browser-reachable code paths unless the route explicitly needs privileged aggregate reads and returns a strict projection.

## Risks and Mitigations

### Risk: Player privacy remains too broad

Current `players` RLS exposes all player columns to everyone. Fix this before adding dashboard APIs.

Mitigation: owner-only direct `players` SELECT plus `public_player_profiles` view.

### Risk: Map code accidentally introduces points

MapLibre examples often use markers and point sources. That pattern conflicts with this project.

Mitigation: centralize GeoJSON generation and test that all territory features are Polygons.

### Risk: API route returns raw Supabase rows

Raw rows can include internal ids like `private_location_id` or future sensitive fields.

Mitigation: route handlers must map rows into explicit response DTOs.

### Risk: OAuth E2E blocks CI

GitHub OAuth is difficult to automate reliably.

Mitigation: unit-test route mappers and auth branching; add Playwright skeleton for redirects; defer full live OAuth E2E until test auth strategy exists.

### Risk: Territory seed becomes throwaway data

Static frontend seed data would make M6 harder.

Mitigation: seed DB-shaped `territory_cells` and consume it through `/api/territory`.

## Open Questions

1. Should `territory_cells` be treated as canonical in M2/3 or as a cache of `territory_events`?

Recommendation: canonical for M2/3 seed state, then maintained by M6 recalculation jobs.

2. Should `/api/territory` be public immediately?

Recommendation: yes, because the payload must be public-safe anyway. The dashboard route can remain authenticated.

3. Should faction choice be editable?

Recommendation: no for the pilot. Lock faction after initial onboarding; add admin reset later.

4. Should display names be unique?

Recommendation: yes, globally and case-insensitively.
