## Review Verdict

Proceed with the explicit staged release contract from the CEO review, with one engineering correction: the deploy script must be the single owner of release state transitions.

The complete staging slice should include:

1. `GET /api/health` with a small, non-secret JSON response.
2. `next.config.ts` set to `output: "standalone"`.
3. Artifact packaging that copies `.next/standalone`, `.next/static`, and `public`.
4. `scripts/deploy/staging-install.sh` that uploads, unpacks, validates, switches `current`, restarts systemd, health-checks, and rolls back on failure.
5. `scripts/deploy/staging-rollback.sh` for explicit operator rollback.
6. Committed systemd and Nginx templates under `deploy/staging/`.
7. GitHub Actions workflow with CI gates, artifact build, artifact transfer, deploy, and external health verification.
8. A staging runbook documenting VPS setup, DNS/TLS, secrets, media paths, rollback, and verification.

Completeness: 10/10. This is the right amount of infrastructure for the next GPS/media milestones; a shortcut here would become operational debt immediately.

- one health route,

- one install script,
- one rollback script,
- one systemd template,
- one Nginx template,
- one GitHub Actions workflow,
- one runbook.

This is more than eight files, but the complexity is justified because each file is a separate operational contract. Combining them would make the deploy harder to audit.

### Search Check

- **[Layer 1] Next.js standalone:** use `output: "standalone"` and package the minimal server, but manually copy `public` and `.next/static` into the standalone directory because Next does not copy those by default.
- **[Layer 1] GitHub Actions environments:** use a `staging` environment so staging secrets are scoped to deploy jobs and can be branch-restricted.
- **[Layer 1] systemd:** use a normal long-running service with one `ExecStart`. Prefer `Type=exec` if available on the VPS systemd version; otherwise `Type=simple` is acceptable with health checks in the deploy script.
- **[Layer 1] Nginx:** proxy app traffic to `127.0.0.1:4201`, preserve forwarded headers, and statically serve only `/media/public/`.

### Architecture Issues

1. **Rollback must be automatic during deploy failure.**

Recommendation: implement rollback inside `staging-install.sh`, not only as a separate operator command.

Reason: the risky moment is after `current` changes and before health is proven. A separate rollback script is useful, but the install path must restore the previous symlink automatically if restart or health checks fail.

2. **Health checks need two levels.**

Recommendation: require local health first, then external health. Local failure means rollback. External failure with local success should fail the workflow but not necessarily roll back the app because DNS/TLS/Nginx may be the broken layer.

Reason: this separates application bad releases from edge/proxy misconfiguration.

3. **Secrets must be runtime-only, not artifact inputs.**

Recommendation: GitHub Actions should use deploy SSH secrets and browser-safe build values only. Server-only values stay in `/opt/meccha-chameleon-staging/shared/.env.production` on the VPS.

Reason: artifact transfer is easier to audit when release tarballs never contain `.env*` or service-role secrets.

4. **Nginx must explicitly deny or avoid private media.**

Recommendation: add only a `/media/public/` alias. Do not add a broad `/media/` alias. Add an explicit `location ^~ /media/private/ { return 404; }` defense if the config has any broader media block.

Reason: the project invariant says proof media never has a public static URL.

## Code Quality Review

### Recommended File Layout

```text
src/app/api/health/route.ts
deploy/staging/meccha-chameleon-staging.service
deploy/staging/nginx-meccha-staging.conf
scripts/deploy/staging-install.sh
scripts/deploy/staging-rollback.sh
scripts/deploy/package-standalone.sh
.github/workflows/staging.yml
docs/deployment-staging.md
```

### Code Quality Issues

1. **Shell scripts need strict mode and explicit inputs.**

Recommendation: all deploy scripts should use `set -Eeuo pipefail`, validate required arguments, and print only release IDs and command status, never environment values.

2. **Artifact package rules should be centralized.**

Recommendation: add `scripts/deploy/package-standalone.sh` instead of embedding tar commands directly in YAML.

Reason: it avoids duplicating Next standalone packaging rules between local verification and GitHub Actions.

3. **Release IDs should be deterministic.**

Recommendation: use `<utc-timestamp>-<short-sha>` and write a `REVISION` file into each release containing full SHA, build time, and workflow run URL.

Reason: operators need to answer "what is live?" without opening GitHub Actions.

4. **Systemd hardening should be included but not overdone.**

Recommendation: include `User=deploy`, `WorkingDirectory=/opt/meccha-chameleon-staging/current`, `EnvironmentFile=/opt/meccha-chameleon-staging/shared/.env.production`, `Environment=NODE_ENV=production PORT=4201 HOSTNAME=127.0.0.1`, `ExecStart=/usr/bin/node server.js`, `Restart=on-failure`, `RestartSec=5`, `NoNewPrivileges=true`, `PrivateTmp=true`.

Reason: this is enough least-privilege posture without introducing sandboxing settings that may break Next/sharp file access.

## Performance Review

No application hot-path performance concerns. The deployment path should avoid unnecessary server work:

- Build in GitHub Actions, not on the VPS.
- Upload a compressed artifact.
- Keep old releases pruned to a fixed count, for example the newest 5 successful releases.
- Do not include full `node_modules`; use Next standalone output.

## Failure Modes

| Failure | Handling required | Test required |
|---------|-------------------|---------------|
| Artifact missing `server.js` | Fail before symlink switch | package validation test |
| `public` or `.next/static` missing | Fail before deploy or smoke test catches broken assets | package test |
| `.env.production` missing on VPS | Fail before symlink switch | install script test |
| `systemctl restart` fails | Restore previous symlink and restart previous release | mocked install test |
| Local health fails | Restore previous symlink and mark workflow failed | mocked install test |
| External health fails | Mark workflow failed; preserve locally healthy release unless configured otherwise | workflow smoke test |
| Nginx exposes private media | Block deploy/runbook verification | Nginx verification |
| Artifact contains `.env*` | Fail packaging | package test |
| Staging uses production media path | Fail runbook verification | install validation |

Critical gaps if omitted: automatic rollback on local health failure, artifact secret exclusion, and private media non-exposure.

## NOT In Scope

- Production deployment automation: staging first; production promotion should be a separate explicit milestone.
- Docker image or container registry: the requested shape is systemd plus immutable release directories.
- Automated Cloudflare DNS/TLS provisioning: document the checklist now; automate later if it becomes repetitive.
- Full Playwright suite: useful after staging exists, but not required to implement the deploy foundation.
- Backups and restore drills for media: necessary before pilot launch, not needed for first staging deploy.
- Background workers: no worker process exists yet.
