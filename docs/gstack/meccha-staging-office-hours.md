## Problem Statement

Meccha Chameleon needs a real staging environment before the next gameplay milestones add GPS submission, image uploads, moderation, proof review, and territory maintenance jobs.

The requested staging slice is:

- Immutable artifact deployment
- systemd service running the Next.js app on port `4201`
- Nginx virtual host for `staging.meccha.fun`
- Cloudflare DNS and TLS in front of the VPS
- Persistent staging media directories under `/srv/meccha-chameleon-staging/media/`
- Health endpoint for deploy and uptime checks
- GitHub Actions workflow for repeatable deployment

This is not product polish. It is operational groundwork so future GPS/media features can be tested in an environment that matches production without risking production data or leaking private coordinates.

## Demand Evidence

This is an infrastructure dependency already implied by the committed plan:

- `docs/delivery-plan.md` requires staging to deploy automatically on `main`.
- `docs/test-strategy.md` says E2E tests run against staging.
- `docs/security-model.md` requires separate staging media paths and forbids Supabase Storage for clue/proof photos.
- `docs/architecture.md` already proposes a VPS deployment architecture and `GET /api/health`, but it still uses a generic Docker-based shape and an old staging hostname.

The pain is concrete: without staging, M4/M5 upload and moderation work can only be tested locally or directly in production-like infrastructure by hand.

## Status Quo

Current repo state:

- Next.js app, Supabase SSR auth, MapLibre territory map, and tests exist.
- No `.github/workflows/` files are present.
- No health route exists yet.
- No deployment scripts, systemd units, or Nginx templates are committed.
- Existing ADR-005 says Docker on VPS, but the requested slice is immutable release directories plus systemd on port `4201`.
- Staging media storage is specified in docs but not provisioned by code or runbook.

## Target User & Narrowest Wedge

The immediate user is the maintainer/operator deploying Meccha Chameleon to a VPS.

The narrowest valuable version is a staging deploy where a push to `main` can produce a versioned release artifact, install it on the VPS, atomically switch the active release, restart `meccha-chameleon-staging.service`, and verify `https://staging.meccha.fun/api/health`.

That is enough to support:

- future E2E tests against staging,
- Cloudflare/Nginx/TLS verification,
- media directory permission checks,
- rollback by switching the release symlink back to a previous artifact.

## Constraints

- Exact coordinates must never reach browser JavaScript, public APIs, logs, map payloads, or client Supabase queries.
- `SUPABASE_SERVICE_ROLE_KEY`, `TURNSTILE_SECRET_KEY`, deployment SSH keys, and Cloudflare API tokens must be server-side or GitHub-secret only. None may use `NEXT_PUBLIC_`.
- Staging and production media directories must be separate:
  - staging: `/srv/meccha-chameleon-staging/media/public/clues/`
  - staging: `/srv/meccha-chameleon-staging/media/private/proofs/`
  - production: `/srv/meccha-chameleon/media/...`
- Media storage must remain outside immutable release directories.
- Public clue media may be served statically by Nginx with long-lived cache headers.
- Private proof media must not be statically served.
- Deployment artifacts must not bake secrets into committed code or release bundles.
- Arch owns Git/GitHub operations. This office-hours artifact is planning only and should be committed by Arch before implementation.

## Premises

1. Staging deployment is the correct next infrastructure wedge because M4/M5 introduce GPS and media flows that need production-like verification before real users.
2. Immutable release directories plus a stable `current` symlink are a better fit than in-place `git pull` deployments because rollback and artifact identity matter more than saving a few shell steps.
3. systemd on port `4201` is sufficient for the current Next.js app; Docker can remain a later option if background workers, image isolation, or multi-service orchestration become necessary.
4. Nginx must own TLS termination/proxying and static public media serving, while the app owns private media authorization.
5. GitHub Actions should deploy staging automatically from `main`, but production promotion should remain explicit and separate.

## Approaches Considered

### Approach A: Minimal systemd deploy

Summary: Build the app on GitHub Actions, upload a tarball to the VPS, unpack it into a timestamped release directory, point `/opt/meccha-chameleon-staging/current` at it, restart systemd, and verify `/api/health`.

Effort: M  
Risk: Low

Pros:

- Smallest deploy surface.
- Easy rollback by repointing `current`.
- Matches the requested port `4201` and VPS/systemd shape.
- Does not require introducing Docker or a registry.

Cons:

- Requires care to package the right Next.js standalone files.
- Node/npm version drift between CI and VPS must be managed.
- Background workers will need separate services later.

Reuses:

- Existing `npm run build`, `npm run lint`, `npm run typecheck`, and `npm test` scripts.
- Existing VPS media path decisions from `docs/security-model.md`.

### Approach B: Ideal staged release with explicit deploy contract

Summary: Add a full deployment contract: health route, standalone Next.js build output, `scripts/deploy/staging-install.sh`, `scripts/deploy/staging-rollback.sh`, committed systemd and Nginx templates, GitHub Actions CI/deploy workflow, and a staging runbook.

Effort: L  
Risk: Low-Med

Pros:

- Best long-term operating model.
- Makes setup reproducible on a fresh VPS.
- Captures rollback, media permissions, health checks, and Cloudflare/Nginx requirements in one place.
- Gives future `/qa` and E2E work a stable target.

Cons:

- More files and docs than the smallest possible deploy.
- Some secrets and VPS facts still need to be configured outside the repo.
- Requires discipline to keep templates in sync with live server config.

Reuses:

- Existing architecture/security/test docs.
- Next.js build pipeline.
- Existing media-storage privacy model.

### Approach C: Container registry deploy

Summary: Build a Docker image in GitHub Actions, push it to a registry, SSH into the VPS, pull the tagged image, and restart a container behind Nginx.

Effort: L  
Risk: Med

Pros:

- Strong artifact isolation.
- Clean path to workers and multi-service deployment.
- Aligns with the older ADR-005 container wording.

Cons:

- Does not match the user's requested systemd-on-port-4201 shape unless systemd manages Docker.
- Introduces registry credentials and Docker operational overhead now.
- Media volume mounts and private media paths still need separate careful handling.

Reuses:

- Existing ADR-005 concept, but would need revision to fit the concrete staging host.

## Recommended Approach

Choose **Approach B: Ideal staged release with explicit deploy contract**.

Reason: this deployment slice is small enough to do completely, and the completeness pays for itself when GPS/media features arrive. Approach A ships fastest, but Approach B prevents the staging environment from becoming a one-off shell history artifact.

The implementation should still sequence like Approach A:

1. Add `/api/health`.
2. Make `next build` produce a deployable standalone artifact.
3. Add systemd and Nginx templates.
4. Add deploy scripts with idempotent directory creation and rollback.
5. Add GitHub Actions CI plus staging deploy.
6. Add runbook docs and verification checklist.

## Required Deployment Shape

Suggested VPS paths:

```text
/opt/meccha-chameleon-staging/
  releases/
    20260629T153000Z-<sha>/
  shared/
    .env.production
  current -> releases/<active-release>

/srv/meccha-chameleon-staging/
  media/
    public/clues/
    private/proofs/
```

Suggested service contract:

- systemd unit: `meccha-chameleon-staging.service`
- working directory: `/opt/meccha-chameleon-staging/current`
- app port: `4201`
- environment file: `/opt/meccha-chameleon-staging/shared/.env.production`
- restart policy: `Restart=on-failure`
- health check: `GET http://127.0.0.1:4201/api/health`

Suggested Nginx contract:

- `server_name staging.meccha.fun`
- proxy app traffic to `127.0.0.1:4201`
- serve public clue media from `/srv/meccha-chameleon-staging/media/public/`
- do not expose `/srv/meccha-chameleon-staging/media/private/`
- preserve `Host`, `X-Forwarded-Proto`, and `X-Forwarded-For`
- set public clue cache headers to `public, max-age=31536000, immutable`

Suggested Cloudflare contract:

- proxied DNS record for `staging.meccha.fun`
- SSL/TLS mode: Full (strict) once origin certificate is installed
- Always Use HTTPS enabled
- private media paths must not be cacheable
- public clue media may be cached using the existing immutable filename strategy

## Open Questions

- Should staging deploy on every push to `main`, or only after CI passes on `main`?
- Should the GitHub Actions deploy job build on GitHub and upload artifacts, or SSH to the VPS and build there? Recommendation: build on GitHub, deploy artifacts.
- Which Linux user owns release files and media directories? Recommendation: `deploy` owns release/media directories; systemd runs the app as a least-privileged app user if available.
- Is Cloudflare already authoritative for `meccha.fun`, and is an origin certificate already installed on the VPS?
- Should staging use a separate Supabase project immediately, or can it start with a test/staging project already configured? Recommendation: separate Supabase staging project before any GPS/media submission flow is tested.

## Success Criteria

- `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build` pass in CI before deploy.
- GitHub Actions can deploy staging from `main` without manual SSH commands.
- `https://staging.meccha.fun/api/health` returns `200` with no secret values.
- The systemd service is enabled, running, and bound only to localhost port `4201`.
- Nginx serves the app over HTTPS and proxies to `127.0.0.1:4201`.
- Public staging media directory exists and is static-servable.
- Private staging media directory exists but is not reachable as a static URL.
- A failed deployment leaves the previous `current` release running.
- Rollback to the previous release is documented and tested once.
- No browser bundle or API payload contains exact coordinates, service role keys, Turnstile secrets, or private media paths.

## Distribution Plan

The deliverable is a deployed web service, not a user-installed artifact.

Distribution path:

1. GitHub Actions checks out source on `main`.
2. CI runs lint, typecheck, tests, and build.
3. Workflow packages the deployable Next.js artifact with a release ID containing timestamp and git SHA.
4. Workflow uploads artifact to the VPS over SSH.
5. VPS install script unpacks into `/opt/meccha-chameleon-staging/releases/<release-id>`.
6. Install script ensures persistent media directories exist under `/srv/meccha-chameleon-staging/media/`.
7. Install script updates `current` symlink atomically.
8. systemd restarts `meccha-chameleon-staging.service`.
9. Workflow verifies `https://staging.meccha.fun/api/health`.

Secrets live in GitHub Actions secrets and on the VPS environment file, not in the artifact:

- `STAGING_DEPLOY_HOST`
- `STAGING_DEPLOY_USER`
- `STAGING_DEPLOY_SSH_KEY`
- `STAGING_DEPLOY_PATH`
- `STAGING_APP_URL`
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `NEXT_PUBLIC_TURNSTILE_SITE_KEY`
- `TURNSTILE_SECRET_KEY`
- optional Cloudflare API token if DNS/cache rules are automated

## Dependencies

- VPS user and SSH key configured for non-interactive deploys.
- Node.js runtime version on VPS compatible with the version used by CI.
- Nginx installed and able to reload via a controlled deploy command or documented manual setup.
- Cloudflare will manage `meccha.fun` DNS (not amfbss.com).
- Staging Supabase project and OAuth redirect URLs configured for the staging hostname.
- `.env.production` created on the VPS with staging-only values.

## What I Noticed

- You specified operational constraints directly: "immutable artifact deployment", "systemd service on port 4201", and the exact staging hostname. That is enough to avoid an abstract deployment plan.
- The request includes persistent media directories, which means you are treating deployment and privacy as one system. That matters for this repo because image storage is part of the security boundary.
- The staging slice is correctly placed before more GPS and upload work. It gives future milestones somewhere realistic to fail before users or production data are involved.
