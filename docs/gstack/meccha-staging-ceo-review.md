## Verdict

Proceed with the staging deployment plan, but treat it as a security and operations milestone, not a hosting chore.

The recommended plan is **Approach B: staged release with explicit deploy contract** from `docs/gstack/meccha-staging-office-hours.md`: Next.js standalone artifact, immutable release directories, systemd on localhost port `4201`, Nginx and Cloudflare in front, separate staging media storage, health checks, rollback script, GitHub Actions deployment, and a runbook.

The important CEO-level correction: staging must prove the privacy model before M4/M5 introduce GPS and image uploads. A staging deploy that cannot verify secret isolation, media separation, private proof non-exposure, and rollback is not complete.

## Premise Challenge

This is the right next problem. Without staging, M4/M5 will force the team to test GPS submission, media writes, EXIF stripping, moderation, and proof privacy locally or against production-like infrastructure by hand.

The actual outcome is not "site available at a staging URL." The outcome is:

- Every push to `main` can produce a repeatable staging release.
- The operator can identify exactly what SHA is live.
- A bad release fails closed or rolls back without damaging media.
- Staging uses separate secrets, separate Supabase, separate media paths, and separate DNS.
- Future E2E and manual QA can verify privacy boundaries before real pilot usage.

Doing nothing keeps deployment as tribal shell history. That is tolerable for a static landing page; it is unacceptable before GPS and private proof photos.

## Existing Code Leverage

Reuse:

- Existing `npm run lint`, `npm run typecheck`, `npm test`, and `npm run build`.
- Existing Supabase server/browser client split.
- Existing Turnstile middleware and credential model.
- Existing EXIF stripping library and VPS media decision.
- Existing H3/map DTO privacy tests as a pattern for deployment privacy checks.

Do not rebuild:

- Do not introduce Docker just to satisfy old ADR-005 wording. The current requested shape is systemd plus immutable release directories.
- Do not build a custom process manager. systemd already provides restart policy, logs, enable-on-boot, and service state.
- Do not place media in the release directory. Releases are disposable; media is durable.

## Alternatives

### Approach A: Minimal systemd deploy

Summary: Build in GitHub Actions, upload a tarball, switch `current`, restart systemd, verify health.

Effort: M  
Risk: Low  
Completeness: 7/10

Pros:

- Fastest path to a live staging URL.
- Easy to understand.
- Avoids Docker/registry overhead.

Cons:

- Easy to leave DNS, TLS, media permissions, rollback, and secrets as undocumented manual steps.
- Future E2E and M4/M5 work may inherit an unreliable staging target.

### Approach B: Staged release with explicit deploy contract

Summary: Add health route, standalone build config, artifact packaging, install/rollback scripts, systemd and Nginx templates, GitHub Actions workflow, environment model, and runbook.

Effort: L  
Risk: Low-Med  
Completeness: 10/10

Pros:

- Makes staging reproducible and auditable.
- Gives rollback and artifact identity from day one.
- Converts privacy/security requirements into deploy-time checks.
- Best foundation for E2E, upload, moderation, and proof flows.

Cons:

- More files and documentation.
- Requires some VPS/Cloudflare/GitHub secrets to be configured outside the repo.

### Approach C: Docker registry deploy

Summary: Build a Docker image, push to registry, pull and restart on VPS.

Effort: L  
Risk: Med  
Completeness: 8/10

Pros:

- Strong isolation boundary.
- Good future path for workers.
- Aligns with old ADR-005.

Cons:

- Does not match the requested systemd-on-port-4201 deployment shape unless systemd manages Docker.
- Adds registry credentials and container operations before the app needs them.
- Still needs the same external media and secret discipline.

## Recommendation

Choose **Approach B**.

This is a boilable lake: the difference between a minimal deploy and a complete deploy contract is small, and the operational leverage is high. The project is about to add GPS and private media; this is the moment to make staging boring, repeatable, and privacy-aware.

## Deployment Contract

```text
GitHub main
    |
    v
CI gates: lint, typecheck, unit tests, build, security greps
    |
    v
Build standalone Next.js artifact
    |
    v
Upload release tarball to VPS
    |
    v
/opt/meccha-chameleon-staging/releases/<timestamp>-<sha>
    |
    v
Atomic current symlink switch
    |
    v
systemd restart: meccha-chameleon-staging.service
    |
    v
Health check: http://127.0.0.1:4201/api/health
    |
    v
External check: https://meccha-staging.amfbss.com/api/health
```

Release paths:

```text
/opt/meccha-chameleon-staging/
  releases/
    20260629T153000Z-<sha>/
  shared/
    .env.production
  current -> releases/<active-release>

/srv/meccha-chameleon-staging/
  media/
    public/clues/
    private/proofs/
```

Runtime contract:

- App listens on `127.0.0.1:4201`.
- systemd service name: `meccha-chameleon-staging.service`.
- Environment file: `/opt/meccha-chameleon-staging/shared/.env.production`.
- Release artifact contains no `.env*` file and no secrets.
- Public clue media is served by Nginx from `/srv/meccha-chameleon-staging/media/public/`.
- Private proof media is not statically served by Nginx.

## Environment Model

Staging must have its own:

- Supabase project.
- Supabase anon key.
- Supabase service role key.
- Turnstile site key and secret key.
- OAuth callback URLs.
- Cloudflare DNS record.
- VPS release path.
- VPS media path.
- GitHub Actions environment named `staging`.

GitHub Actions secrets:

- `STAGING_SSH_HOST`
- `STAGING_SSH_USER`
- `STAGING_SSH_KEY`
- `STAGING_SSH_PORT`
- `STAGING_BASE_URL`
- `STAGING_SUPABASE_URL`
- `STAGING_SUPABASE_ANON_KEY`
- Optional deploy-only Cloudflare token if DNS/TLS provisioning is automated.

VPS-only secrets in `/opt/meccha-chameleon-staging/shared/.env.production`:

- `SUPABASE_SERVICE_ROLE_KEY`
- `TURNSTILE_SECRET_KEY`
- Any future server-only signing or storage secrets.

Browser-safe values may be present in the runtime env only if already allowed by `docs/security-model.md`, such as `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `NEXT_PUBLIC_TURNSTILE_SITE_KEY`, and map style variables.

## DNS and TLS

DNS/TLS should be provisioned as an explicit checklist, not hidden in a deploy script.

Required:

- Cloudflare proxied DNS record: `meccha-staging.amfbss.com`.
- Nginx virtual host for `meccha-staging.amfbss.com`.
- Origin certificate installed on the VPS before enabling Full strict.
- Cloudflare SSL/TLS mode: Full strict.
- Always Use HTTPS enabled.
- Public clue media cache rule only for public media path.
- Private media path bypass/no-store if a private route is ever introduced under a cacheable hostname.

Important source checks:

- Next.js standalone output supports deploying `.next/standalone` without installing full `node_modules`, but `public` and `.next/static` must be copied into the standalone directory if the minimal server serves them.
- Cloudflare Full strict requires a valid unexpired origin certificate for the requested hostname.
- GitHub Actions environments can scope secrets/variables and restrict which branches may deploy.

## Rollback Strategy

Rollback must be tested during the staging milestone, not documented for later.

Rollback command shape:

```text
staging-rollback.sh [previous|<release-id>]
  verify target release exists
  switch current symlink atomically
  restart systemd
  check local health
  check external health
  leave failed rollback with previous current restored
```

Failure policy:

- If artifact upload fails, current release remains untouched.
- If unpack or dependency validation fails, current release remains untouched.
- If systemd restart fails after symlink switch, rollback to prior symlink immediately.
- If local health fails, rollback immediately.
- If external health fails but local health passes, keep service up but mark deploy failed for DNS/TLS investigation unless the app release itself is clearly bad.
- Media directories are never rolled back with app releases.

## Failure Modes

| Failure | Impact | Required handling |
|---|---|---|
| CI fails | No deploy | Block deployment before artifact creation |
| Artifact missing static files | Broken CSS/assets | Build packaging test checks `.next/static` and `public` copied |
| Secrets baked into artifact | Secret leak | Artifact grep and no `.env*` packaging rule |
| Wrong Supabase project | Cross-environment data leak | Health route reports non-secret environment name; runbook verifies staging project URL |
| Service binds public interface | App bypasses Nginx/Cloudflare | systemd uses `HOSTNAME=127.0.0.1`; deploy verifies local bind |
| Nginx exposes private media | Proof photo leak | Config test and explicit negative `curl /media/private/...` check |
| Cloudflare Flexible mode | Origin traffic not strictly verified | TLS checklist requires Full strict before done |
| Failed release switches current | Staging outage | Deploy switches symlink only after release validates |
| Rollback script targets missing release | Longer outage | Script validates release dir before mutation |
| Public cache applied too broadly | Private content cached | Cache rule limited to public clue path only |

## Error and Rescue Map

```text
deploy job
  |
  +-- CI error
  |     rescue: fail workflow, no SSH deploy
  |
  +-- SSH/auth error
  |     rescue: fail workflow, current untouched
  |
  +-- remote install error
  |     rescue: remove incomplete release, current untouched
  |
  +-- symlink switch error
  |     rescue: current remains prior release or is restored
  |
  +-- systemd restart error
  |     rescue: restore previous current, restart previous release
  |
  +-- local health error
  |     rescue: restore previous current, restart previous release
  |
  +-- external health error
        rescue: report DNS/TLS/proxy failure, preserve local service state
```

## Acceptance Criteria

- `https://meccha-staging.amfbss.com/api/health` returns `200`.
- Health response contains no secrets, coordinates, internal file paths, or raw env dumps.
- GitHub Actions can deploy from `main` without manual SSH commands.
- Failed deploy leaves prior release running.
- Rollback to previous release is tested.
- systemd app process is bound only to localhost port `4201`.
- Nginx proxies app traffic and serves only public media statically.
- Private media directory is not accessible by static URL.
- Staging and production media paths are separate.
- Staging and production Supabase projects are separate before GPS/media flows are tested.
- CI includes lint, typecheck, unit tests, build, and security grep checks.
